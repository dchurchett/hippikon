Hippikon is a powerful authorization API that was developed for enterprise software applications and is now provided free under the GNU LGPL. In other words, what was once proprietary software developed for a highly complex enterprise application where security was paramount is now a free open source project!

Hippikon provides features not found in existing APIs, such as Sun's JAAS. It is an XML driven mechanism that allows static permission rules to be defined, and also allows object state, object hierachy, user associations, permission inheritance and permission overrides into account at runtime. The API is small, fast and simple to use. The design follows a logical domain model based on a number of standard security design patterns.

Highlights

  • May be used in any Java application and is agnostic of frameworks such as J2EE, Spring, Struts etc,
  • XML configuration file defines static permissions and valid object hierachy
  • A simple plugin/callback mechanism is provided to configure permissions for object instance at runtime
  • A single instance may configure multiple products or applications
  • Decoupled from authentication mechanisms allowing any to be used (such as basic password, LDAP, Digital Certificates etc,)
  • Easy API to learn and use

What Does Hippikon Do?

The Hippikon security model makes a clear separation of concerns between the two primary aspects of an overall security mechanism for a software system:
  • Authentication: Is the user who they say they are?
  • Authorization: What can an authenticated user do in a system?
Hippikon's focus is in providing a convenient object oriented API for handling authorization that encapsulates a set of permissions for a user accessing various objects in a system. The granularity of Hippikon is purely based on objects, and not static pieces of code or object methods. A typical scenario for securing a java application involves first authenticating a user then determing authorization rules within the system i.e., what can they do within the system, what data can they read?, what actions can they perform?

If your application uses a simple role based system where one user may only have one role, then Hippikon may not provide much benefit. However, if you find yourself writing a system where:

  • a user can have multiple roles or,
  • a user can have multiple associations with an object (e.g., the creator, the maintainer, the owner), or
  • there is a need to take the object's internal state into account (e.g., if immutable turn off read access), or
  • the object hierachy must be taken into account (e.g., a note belongs to a document, which is immutable), or
  • a combination of all the above
things will get out of hand very quickly and the code ends up being littered with complex conditional checks. Here is where Hippikon will simplify the development effort tremendously.
